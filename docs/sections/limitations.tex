\section{Limitations}
We tested the solvers and discretization methods on a simplified 1D-model. This does not reflect real-world scenarios with complex three-dimensional geometries where a finite-element approach becomes practically necessary. Furthermore, we did not look deeply into the low-level solver logs to identify the causes of the observed behaviours, nor did we check the exact internal solver settings once initialized via Python, which may in particular explain the unexpected CVODE behaviour when provided with analytically derived Jacobians.

We did not implement any universally robust error calculation scheme. The code, in its default settings, relies on CVODE with very strict \(\texttt{rtol} = \texttt{atol} = 10^{-13}\) tolerances. While the codebase allows for using a different reference solver, none is universally robust without careful configuration. We have also not enabled error checking for most of the benchmarks, as they would be exceedingly numerically expensive.
